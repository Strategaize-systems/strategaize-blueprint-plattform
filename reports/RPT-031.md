# Requirements V3.1 — Mirror Usability

## Outcome Summary

V3.1 Requirements vollständig definiert. 3 Features, 4 Backlog-Items, Roadmap + PRD aktualisiert.

## V3.1 Scope

### Problem

V3 Phase 1 liefert die Mirror-Infrastruktur, aber für den realen Einsatz fehlt:
- GF muss Teilnehmer-Infos extern liefern (kein Formular in der Plattform)
- Einladungs-E-Mail ist generisch (Teilnehmer kennen Strategaize nicht)
- Kein Zeitrahmen auf Runs (Teilnehmer wissen nicht bis wann)

### Features

|----|---------|-----------|-------------|
| FEAT-032 | GF Mirror-Nominierungsformular | P0 | Tenant-Admin schlägt Teilnehmer vor (Name, Ebene, Abteilung, E-Mail). Admin lädt ein. |
| FEAT-033 | Verbessertes Mirror-Onboarding | P0 | Eigenes E-Mail-Template mit Kontext. Erweiterte Policy-Seite mit Erklärung + Video-Platzhalter. |
| FEAT-034 | Mirror-Run Deadline | P1 | due_date auf Runs. Dashboard-Anzeige mit visuellem Hinweis bei nahender Deadline. |

### Out of Scope (V3.1)

- Direktkommunikation Strategaize ↔ Teilnehmer (explizit ausgeschlossen)
- In-App Messaging
- Automatische Reminder-E-Mails (V3.2+)
- Bulk-Import für Nominations (V3.2+)
- Video-Produktion (nur Platzhalter)
- Cross-System-Integration mit Business System / "Mein Tag" (separates Discovery)
- target_roles DB-Filterung (separate Kataloge statt Systemkomplexität)
- question_type Single-Choice UI (Optionen im Fragetext)

### Strategische Entscheidungen (dokumentiert)

1. Blueprint bleibt "dummer Konsument" — JSON rein, Fragen zeigen, Antworten sammeln, ZIP raus
2. Intelligence Studio wird die Katalog-Fabrik (Skills für Erstellung, Lückenanalyse, Überarbeitung)
3. "Mein Tag" im Business System ist das einzige Zeit/Prioritäts-Cockpit
4. Separate Kataloge pro Block + Ebene statt DB-Filterung
