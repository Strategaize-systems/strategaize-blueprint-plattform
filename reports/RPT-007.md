## 5. Wiring-Plausibilität

### 5.1 Auth-Flow

```
LoginForm → login() Server Action → createClient() (internal URL) → GoTrue → Session Cookie → Middleware validates → Dashboard
```
- Rate-Limiting vor Auth-Call ✓
- Internal URL für Server-Side ✓
- Cookie-basierte Session ✓

### 5.2 Admin-Tenant-Flow

```
TenantsClient → fetch(/api/admin/tenants) → requireAdmin() → adminClient.from("tenants") → Batch enrichment → JSON response → UI render
```
- Auth-Check (requireAdmin) ✓
- service_role mit GRANTs ✓
- Batch-Queries statt N+1 ✓

### 5.3 Tenant-Question-Event-Flow

```
RunWorkspaceClient → fetch(/api/tenant/runs/[id]/questions/[qid]/events) POST → requireTenant() → RLS-filtered INSERT → client_event_id idempotency → 201 Created
```
- Tenant-Isolation via RLS ✓
- Idempotency via client_event_id ✓
- Run-Lock-Check ✓
- Append-only (RLS + DB-Trigger) ✓

### 5.4 Evidence-Upload-Flow

```
RunWorkspaceClient → fetch(/api/tenant/runs/[id]/evidence) POST (multipart) → requireTenant() → file validation → SHA256 → Storage Upload FIRST → DB INSERT with file_path → evidence_link + event
```
- Upload-First: kein verwaister DB-Record bei Upload-Fehler ✓
- Cleanup: Storage-File gelöscht bei DB-Insert-Fehler ✓
- tenant_id auf evidence_links ✓
- Append-only kompatibel (kein UPDATE nötig) ✓

### 5.5 Evidence-Links Cross-Tenant-Schutz

```
evidence_links INSERT → RLS Policy prüft:
  1. evidence_item_id gehört eigenem Tenant ✓
  2. Run nicht locked ✓
  3. link_type='run' → Run gehört eigenem Tenant ✓
  4. link_type='question' → Question erreichbar über eigenen Run ✓
```

**Ergebnis: Wiring ist plausibel und konsistent mit Architektur.**
