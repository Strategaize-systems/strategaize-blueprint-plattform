## Findings

### Medium

1. **Dupliziertes CatalogQuestion Interface**
   - `src/lib/llm.ts:289`: `interface CatalogQuestion { frage_id, block, unterbereich, fragetext }` (4 Felder, nicht exportiert)
   - `src/lib/freeform.ts:34`: `export interface CatalogQuestion { id, frage_id, block, ebene, unterbereich, fragetext, position }` (7 Felder, exportiert)
   - Funktional kein Problem (structural typing — die 7-Felder-Version erfüllt das 4-Felder-Interface). Aber duplizierte Definitionen sind wartungsgefährlich.
   - **Empfehlung:** In SLC-054 (API-Routen) die llm.ts-Version auf den Import aus freeform.ts umstellen: `import type { CatalogQuestion } from "@/lib/freeform"` und die 4 Felder als Pick verwenden, oder das llm.ts Interface entfernen und die Funktionen die volle Struktur akzeptieren lassen.

2. **RLS-Pattern-Inkonsistenz in Migration 019**
   - Migration 019 nutzt Subquery `(SELECT p.tenant_id FROM profiles p WHERE p.id = auth.uid())` statt der vorhandenen Hilfsfunktion `auth.user_tenant_id()`.
   - `auth.user_tenant_id()` existiert in `sql/rls.sql` und macht exakt denselben Query.
   - Migration 018 (V3.1) nutzt `auth.user_tenant_id()`.
   - Funktional identisch, aber stilistisch inkonsistent.
   - **Empfehlung:** Migration-Datei anpassen bevor sie auf Hetzner ausgeführt wird.

### Low

3. **Theoretische Race-Condition in appendMessages()**
   - `freeform.ts:120-155`: Read-modify-write Pattern (liest messages, appended, schreibt zurück).
   - In der Praxis ausgeschlossen: Free-Form Chat ist 1:1 (ein User wartet auf LLM-Antwort bevor nächste Nachricht).
   - Kein Fix nötig — nur dokumentiert.

4. **parseMappingResult() gibt leeres Array bei Fehler zurück**
   - `freeform.ts:292`: `catch { return []; }` — stille Fehlerbehandlung.
   - Akzeptabel weil der Aufrufer (mapping-Route in SLC-054) das leere Array als "keine Mappings gefunden" behandeln muss und entsprechend reagieren kann.
   - Logging könnte in SLC-054 auf Route-Ebene ergänzt werden.
